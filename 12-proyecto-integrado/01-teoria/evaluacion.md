# Evaluación del Proyecto Final — PIA

## Resumen

La evaluación del proyecto final valora que el equipo construya una solución funcional, documentada y defendible, basada en varias fuentes de datos integradas y con un componente de IA/ML justificado.

No se evalúa solo que “funcione algo”. Se evalúa que exista una relación clara entre problema, datos, integración, modelo, salida útil y decisiones técnicas.

## Distribución orientativa

Esta distribución corresponde al **proyecto final de módulo**. No modifica la evaluación interna de UD7, donde el proyecto de stack convergente pesa el 40% de la unidad.

| Bloque | Peso | Qué se valora |
|---|---:|---|
| Integración de datos | 20% | Uso de al menos dos fuentes, integración real, limpieza y trazabilidad |
| Componente IA/ML | 25% | Modelo, sistema inteligente, RAG, recomendador, clasificación, predicción o análisis justificado |
| Ingeniería del proyecto | 20% | Estructura del repositorio, reproducibilidad, código modular, pipeline o automatización |
| Salida útil | 15% | API, dashboard, informe, asistente, ranking, alerta o interfaz con utilidad clara |
| Documentación | 10% | README, explicación de datos, decisiones, limitaciones e instrucciones de uso |
| Defensa oral | 10% | Claridad, evidencias, demo y capacidad para justificar decisiones |

## Requisitos mínimos para aprobar

- [ ] El proyecto ha sido validado previamente por el profesor.
- [ ] Usa al menos dos fuentes de datos diferenciadas.
- [ ] La integración entre fuentes aporta valor real.
- [ ] Incluye limpieza, curación o transformación de datos.
- [ ] Contiene un componente de IA/ML razonado.
- [ ] Produce una salida útil para usuario, análisis o toma de decisiones.
- [ ] Incluye documentación suficiente para ejecutar o entender el proyecto.
- [ ] El equipo realiza una defensa oral con demostración o evidencias.

## Criterios por nivel

| Nivel | Descripción |
|---|---|
| Excelente | Integra varias fuentes con criterio, el modelo aporta valor claro, la solución es reproducible y la defensa justifica decisiones y limitaciones. |
| Bien | Cumple los requisitos mínimos, funciona de forma razonable y está documentado, aunque con margen de mejora técnica. |
| Mejorable | Hay idea válida, pero integración, modelo, documentación o salida útil son débiles o incompletos. |
| Insuficiente | Es un notebook aislado, usa una sola fuente sin integración, copia un ejemplo externo sin adaptación o no demuestra un componente IA/ML claro. |

## Errores frecuentes

- Usar un único CSV y presentarlo como proyecto completo.
- Dividir artificialmente un mismo dataset para simular varias fuentes.
- Hacer solo un dashboard descriptivo sin componente IA/ML.
- Usar un chatbot o LLM sin datos propios ni evaluación.
- Copiar un notebook de Kaggle sin adaptación al problema.
- Añadir muchas herramientas sin integrarlas de verdad.
- No poder explicar por qué se eligió un modelo, una fuente o una arquitectura.

## Devolución temprana de propuestas

Una propuesta puede devolverse antes de aprobarse si:

- no identifica claramente el problema;
- no tiene fuentes de datos viables;
- usa una sola fuente sin integración real;
- el componente IA/ML es decorativo;
- la dificultad es demasiado baja para un proyecto final;
- el alcance es tan grande que no se puede completar en el tiempo disponible.

## Recomendación

Es preferible una solución pequeña, integrada y bien defendida que una arquitectura muy ambiciosa sin cerrar.

El objetivo es demostrar criterio técnico: saber qué datos usar, cómo integrarlos, qué modelo o sistema aplicar, cómo evaluar el resultado y cómo explicar sus límites.
